1. Introducción

Propósito, alcance, términos y documentos relacionados del IDS.

1.1 Propósito del documento

El presente Informe de Diseño de Software (IDS) describe las decisiones técnicas y arquitectónicas que guiarán la construcción del sistema Renta Movil Location. Está elaborado con base en el SRS del proyecto, el documento de Selección de Framework y el documento de Selección y Configuración de Herramientas de Desarrollo. Su objetivo es servir de guía técnica completa para el equipo de desarrollo durante la fase de construcción.

1.2 Alcance del sistema

Renta Movil Location es una plataforma integral de alquiler de vehículos compuesta por una aplicación web responsiva compatible con Chrome 146 y Firefox 134, y una aplicación móvil nativa para Android 13 y 14. El sistema cubre los siguientes procesos: registro y autenticación de usuarios con roles diferenciados (cliente, administrador, operador y supervisor), gestión de catálogo de vehículos con disponibilidad en tiempo real, reservas, pagos a través de Wompi (PSE, tarjetas crédito/débito, Nequi), generación de contratos en PDF, panel administrativo para gestión de flota, reportes en Word, PDF y Excel, auditoría del sistema y notificaciones. El sistema no incluye facturación electrónica con validez fiscal, integración contable avanzada, versión para iOS ni funcionalidad offline.

2. Descripción General del Sistema

Objetivo, usuarios, funcionalidades y contexto del sistema.

2.1 Objetivo del sistema

Desarrollar e implementar un sistema integral de alquiler de vehículos que optimice los procesos de reserva, contratación, gestión de flotas, pagos y generación de reportes, garantizando seguridad, eficiencia y accesibilidad tanto para clientes como para administradores, con el fin de mejorar la experiencia del usuario y fortalecer la competitividad de las empresas del sector.

2.2 Usuarios del sistema

2.4 Contexto del sistema

El sistema opera en dos plataformas: una aplicación web responsiva y una aplicación móvil nativa para Android 13 y 14. Ambas consumen la misma API REST del backend mediante protocolo HTTPS con autenticación JWT. La base de datos relacional PostgreSQL 17 almacena todas las entidades del sistema. Los pagos se procesan en tiempo real a través de la pasarela Wompi mediante webhooks. El control de versiones y CI/CD se gestiona con GitHub y GitHub Actions mediante una estrategia de ramas feature → dev → qa → staging → main.

3.3 Estrategia de ramas y CI/CD

El proyecto usa una estrategia de ramas por ambientes con GitHub Actions para integración y despliegue continuos:

9. Seguridad del Sistema

Autenticación JWT, protección de datos y OWASP Top 10.

9.1 Autenticación y autorización

  • JWT (JSON Web Token): todos los endpoints de la API REST requieren un token JWT válido en el encabezado Authorization: Bearer {token}.
  • Roles diferenciados: cliente, administrador, operador y supervisor. Cada rol tiene acceso únicamente a los endpoints y vistas que le corresponden.
  • Expiración de tokens: los tokens JWT tienen un tiempo de expiración definido. Al expirar, el usuario debe iniciar sesión nuevamente.
  • Almacenamiento seguro del token: en la app móvil se usa Expo SecureStore; en la web se gestiona mediante estado protegido en Zustand.

9.2 Protección de datos y comunicaciones

  • Cifrado SSL/TLS en todas las comunicaciones entre el cliente y el servidor (protocolo HTTPS obligatorio).
  • Las contraseñas se almacenan en la base de datos únicamente en formato hash (bcrypt), nunca en texto plano.
  • Los datos bancarios de Wompi se gestionan mediante tokenización; el sistema nunca almacena números de tarjetas.
  • Los registros de auditoría se cifran en la base de datos para proteger la integridad del historial de acciones.

11. Consideraciones de Implementación

Estándares, estructura, patrones, errores, pruebas y anti-patrones.

11.1 Estándares de codificación

  • Lenguaje: TypeScript en ambas plataformas (web y móvil) para detección de errores en tiempo de desarrollo.
  • Formato: Prettier con configuración compartida (.prettierrc): comillas simples, punto y coma, 2 espacios de sangría, ancho máximo de 80 caracteres.
  • Calidad de código: ESLint con reglas para React y React Native activado en VS Code con validación automática.
  • Convención de nombres: componentes en PascalCase, funciones y variables en camelCase, constantes en UPPER_SNAKE_CASE, archivos de componentes en PascalCase.tsx.
  • Commits: mensajes en español con prefijo descriptivo (feat:, fix:, refactor:, docs:, test:).
  • Sangría y autoguardado: 2 espacios, formateo automático al guardar activado en VS Code.

11.2 Estructura del proyecto

  • App móvil: arquitectura modular en carpetas app/ (rutas Expo Router), components/, modules/, store/, services/, hooks/.
  • App web: arquitectura modular en src/modules/, src/components/, src/store/, src/services/, src/hooks/, src/routes/.
  • Cada módulo contiene: components/ (presentación), hooks/ (lógica), services/ (API), types/ (interfaces TypeScript).
  • Variables de entorno: archivos .env por ambiente (development, qa, staging, production) con prefijos EXPO_PUBLIC_ en móvil y VITE_ en web. Nunca se suben al repositorio.

El proyecto Renta Movil Location es un sistema de tipo web y móvil.   • Capa de presentación web: React 19 + Vite + Tailwind CSS

Lenguajes de programación propuestos

  • Frontend web y móvil: TypeScript
  • Backend y API REST: Java 21 LTS

Anti-patrones a Evitar

Prácticas incorrectas que el equipo debe evitar durante el desarrollo.

Frontend

  • No hacer prop drilling excesivo: los datos compartidos entre múltiples pantallas van en Zustand.
  • La lógica de negocio y las llamadas a la API no van dentro de los componentes visuales sino en services/ y custom hooks.
  • Los estilos de Tailwind CSS son exclusivos de la app web; en la app móvil se usa StyleSheet o NativeWind.
  • localStorage no existe en React Native; el almacenamiento seguro del token JWT se hace siempre con Expo SecureStore.

Backend Java

  • Nunca construir consultas SQL concatenando datos del usuario; siempre usar los métodos de Spring Data JPA o consultas con parámetros nombrados.
  • Las entidades JPA no se exponen directamente en los endpoints REST; se usan DTOs.
  • La lógica de negocio no va en los controladores sino en la capa de servicios.
  • Contraseñas, URLs y claves secretas nunca se escriben directamente en el código; van en variables de entorno o en application.properties excluido del repositorio.

Ambas plataformas

  • No hacer commits directos a main o staging; todo cambio pasa por pull request con revisión aprobada.

1. Comprensión del Diseño

Interpretación de diagramas, elementos indispensables y riesgos del diseño.

Pregunta 1: ¿Cómo interpreta los diagramas del diseño (UML, arquitectura) para iniciar la construcción del software?

Para el proyecto Renta Móvil, interpretamos los diagramas UML de casos de uso identificando los actores principales definidos en el SRS: cliente, administrador, operador, supervisor e invitado. Cada caso de uso (22 en total, desde 'Registrarse' hasta 'Generar tokens / API') nos indica qué funcionalidad debe construirse y quién la usa.

El diagrama de arquitectura N capas nos orienta sobre cómo separar: la capa de presentación (PWA para Android 13/14 y navegadores Chrome 146 y Firefox 134), la capa de lógica de negocio (API REST con control de versiones v1, v2) y la capa de datos (base de datos relacional). Esta separación guía el inicio de la construcción de cada módulo de forma independiente.

Pregunta 2: ¿Qué elementos del diseño son indispensables antes de comenzar a programar?

Antes de programar Renta Móvil es indispensable contar con:

  • Los 71 requerimientos funcionales distribuidos en los 10 módulos definidos en el SRS (Gestión de Usuarios, Gestión de Vehículos, Auditoría y Comunicación, Gestión del Perfil, Atención y Reportes, Administración del Sistema, Validaciones y Calificaciones, API REST, Aplicación Móvil, y Seguridad y Personalización).

  • Los bosquejos de interfaz en Figma para web y móvil, que muestran la distribución de pantallas por sesión (autenticación, catálogo, reserva, pagos, soporte, panel administrativo, etc.).
  • La definición de la pasarela de pagos Wompi (PSE, tarjetas crédito/débito, Nequi) con integración vía webhook.
  • La arquitectura de base de datos relacional y los endpoints REST documentados.

Pregunta 4: ¿Qué riesgos existen si se inicia la construcción sin un diseño completo?

Para Renta Móvil, iniciar sin el diseño completo implica riesgos concretos sobre el proyecto:

Pregunta 7: ¿Qué criterios utilizaría para priorizar los módulos a desarrollar?

Los criterios de priorización para Renta Móvil son:

  • Dependencia funcional: el módulo de autenticación (RF1–RF5) y la API REST base (RF34, RF35) son pre-requisitos de todos los demás módulos.
  • Impacto en el flujo principal: los módulos de reserva (RF11), contrato (RF12), pago (RF13) y vehículos (RF6–RF10) representan el flujo central del negocio y tienen prioridad alta según el SRS.
  • Seguridad: la autenticación JWT con roles y el cumplimiento OWASP Top 10 (RNF4) deben implementarse desde el inicio porque protegen todos los endpoints.
  • Validaciones previas al uso: la validación de licencia (RF32) es prerrequisito para que el cliente pueda reservar, por lo tanto se prioriza antes que la calificación o los reportes.
  • Prioridad declarada en el SRS: los casos de uso marcados como prioridad 'Alta' (registrarse, iniciar sesión, buscar vehículo, reservar, firmar contrato, pagar, gestionar vehículos, etc.) se desarrollan antes que los de prioridad 'Media' o 'Baja'.

Pregunta 8: ¿Cómo estimaría el tiempo requerido para cada actividad de desarrollo?

Para estimar el tiempo en Renta Móvil consideraríamos:

  • La complejidad de cada requerimiento: requerimientos con múltiples subcomponentes como RF11 (16 sub-requerimientos de reserva) o RF25 (5 tipos de reportes del administrador) requieren más tiempo que requerimientos simples como RF65 (3 sub-requerimientos de personalización).
  • Las integraciones externas: la integración con Wompi mediante webhooks (RF13, RF38, RF48, RF63) requiere tiempo adicional para pruebas de transacción y validación de estados de pago.
  • Los criterios de aceptación del SRS: cada requerimiento funcional tiene criterios específicos que deben cumplirse; a mayor número de criterios, mayor tiempo de desarrollo y prueba.
  • La compatibilidad multiplataforma: dado que el SRS requiere que el sistema funcione en web (Chrome 146/Firefox 134) y móvil (Android 13/14 mediante PWA), cada módulo requiere tiempo adicional para pruebas en ambos entornos.
  • Los requerimientos no funcionales: garantizar tiempos de respuesta máximos de 3 segundos (RNF1) y disponibilidad del 99,5% (RNF6) implica tiempo de optimización y configuración de servidor.

  • Probar la compatibilidad en los entornos definidos: Chrome 146, Firefox 134 (web) y Android 13/14 mediante PWA (móvil), según RNF3.
  • Verificar el cumplimiento de accesibilidad WCAG 2.1 nivel AA (RNF10): contraste mínimo 4.5:1, elementos interactivos con etiquetas descriptivas, texto legible desde 14px.
  • Probar los flujos alternos de cada caso de uso: por ejemplo, verificar que tras 3 intentos fallidos de inicio de sesión la cuenta se bloquee (RF2.5, RF43.5).

Actividad 2 — Recomendación de Herramienta de Versionamiento

Empresa con 8 desarrolladores, trabajo remoto, desarrollo simultáneo de módulos e integración continua.

¿Qué herramienta de versionamiento recomendaría?

Las herramientas de versionamiento que recomendamos son Git, usando GitHub o GitLab como plataforma en la nube.

¿Por qué considera que es la mejor opción?

Porque Git fue diseñado exactamente para este tipo de escenario: equipos distribuidos trabajando al mismo tiempo sobre un mismo proyecto. Permite que cada desarrollador tenga su propia copia del código, trabaje de forma independiente en su módulo y luego integre sus cambios sin afectar el trabajo de los demás. Además, es el estándar de la industria, tiene enorme documentación y soporte, y se conecta fácilmente con herramientas de integración continua.

¿Qué ventajas aportaría al equipo de desarrollo?

  • Trabajo en paralelo sin conflictos: cada desarrollador trabaja en su propia rama y los cambios se fusionan de forma controlada.
  • Historial completo de cambios: se sabe quién modificó qué, cuándo y por qué, lo que facilita detectar errores y revertirlos.
  • Integración continua fluida: GitHub Actions o GitLab CI permiten ejecutar pruebas automáticas cada vez que se sube código nuevo.
  • Revisión de código entre compañeros: mediante Pull Requests, otro desarrollador revisa los cambios antes de aprobarlos.
  • Acceso remoto seguro: cualquier miembro del equipo puede trabajar desde cualquier lugar sin depender de un servidor físico.

Actividad 5 — Buenas Prácticas de Codificación en Java

Siete buenas prácticas aplicadas al lenguaje Java.

1. Nombres claros y descriptivos

Las variables, métodos y clases deben decir exactamente qué hacen, sin abreviaciones confusas.

// ❌ Mal
int x = 25;
void calc() {}

// ✅ Bien
int edadUsuario = 25;
void calcularDescuento() {}

2. Una clase, una responsabilidad

Cada clase debe tener un único propósito. Si una clase hace demasiadas cosas, es momento de dividirla.

// ✅ Bien
class Factura { ... }     // Solo maneja facturas
class EnvioCorreo { ... } // Solo maneja correos

3. Evitar métodos muy largos

Un método que hace demasiado es difícil de leer y mantener. Si supera las 20–30 líneas, considera dividirlo.

// ✅ Bien: métodos cortos y concretos
void validarUsuario() { ... }
void guardarUsuario() { ... }
void notificarUsuario() { ... }

4. Usar constantes en lugar de números mágicos

Los números sin contexto confunden a cualquiera que lea el código después.

// ❌ Mal
double impuesto = precio * 0.19;

// ✅ Bien
final double IVA = 0.19;
double impuesto = precio * IVA;

5. Manejo adecuado de excepciones

No ignorar los errores ni capturar todo con un Exception genérico sin dar información útil.

// ❌ Mal
try { ... } catch (Exception e) {}

// ✅ Bien
try { ... } catch (IOException e) {
  System.out.println("Error al leer el archivo: " + e.getMessage());
}

6. Comentar solo lo necesario

El código bien escrito se explica solo. Los comentarios deben aclarar el por qué, no el qué.

// ❌ Innecesario
int edad = 18; // asigna 18 a edad

// ✅ Útil
// La edad mínima requerida por regulación legal
int edadMinima = 18;

7. Aplicar el principio DRY (Don't Repeat Yourself)

Si el mismo bloque de código aparece más de una vez, conviértelo en un método reutilizable.

// ✅ Reutilizable
double aplicarDescuento(double precio, double porcentaje) {
  return precio - (precio * porcentaje);
}

4. Manejo de Errores

4.1 Reglas generales:

  • Toda llamada a la API (Axios) debe estar envuelta en try/catch.
  • Nunca mostrar mensajes técnicos al usuario (ej: '500 Internal Server Error'). Siempre traducir a lenguaje claro.
  • Todo componente que hace una petición debe manejar tres estados: cargando, con datos y con error.

4.2 Estructura estándar de un servicio:

// vehiculosService.js
export const getVehiculos = async () => {
  try {
    const respuesta = await axios.get('/api/v1/vehiculos');
    return respuesta.data;
  } catch (error) {
    console.error('Error al obtener vehículos:', error);
    throw error;
  }
};

EstadoQué mostrar al usuario

Con datosLa información solicitada correctamente.

4.4 Manejo del token JWT:

  • En la app móvil: el token se almacena en Expo SecureStore. Nunca en AsyncStorage sin cifrar.
  • En la app web: el token se gestiona desde el store de Zustand. No se guarda en localStorage.
  • Si el token expira (60 minutos), el sistema redirige al login y muestra: 'Tu sesión ha expirado. Inicia sesión de nuevo.'
  • Todos los endpoints protegidos deben incluir el header: Authorization: Bearer {token}
